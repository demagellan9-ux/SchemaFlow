# Supabase Layer — SchemaFlow

> Global rules are in the root `CLAUDE.md`. This file adds Supabase-specific guidance only.
> For detailed JSONB schema design procedures, invoke `.claude/skills/data-modeling/SKILL.md`.

---

## Responsibilities

Supabase owns:
- Relational metadata storage (projects, uploads, schemas, transformations, jobs)
- Object storage for raw uploaded files and processed output files
- User authentication (Supabase Auth, JWT issuance)
- Row-level security enforcement
- Real-time job status subscriptions (optional — for live progress UI)

Supabase **never**:
- Executes transformation logic
- Parses spreadsheet content
- Stores raw file contents in PostgreSQL columns (binaries belong in Storage)

---

## Core Tables

```sql
users          -- mirrors auth.users; extended profile only
projects       -- user-scoped containers for a consolidation task
uploads        -- one row per uploaded file; references Storage object path
schemas        -- destination schema definition (JSONB column_definitions)
mappings       -- source→destination column mapping per upload (JSONB)
transformations -- ordered transformation rules per destination column (JSONB)
jobs           -- ETL job execution records
job_files      -- per-file status within a batch job
```

---

## Schema Design Rules

- **Stable tables only.** Never `ALTER TABLE` to accommodate a new user schema layout. Schema variations are data, not DDL.
- **JSONB for flexible metadata.** Use `jsonb` columns for: column definitions, transformation rule chains, mapping configurations, validation rule sets, and profiling results.
- **Relational for stable facts.** Use typed columns for: IDs, timestamps, status enums, foreign keys, and counts.
- All tables have `created_at TIMESTAMPTZ DEFAULT now()` and `updated_at TIMESTAMPTZ DEFAULT now()` with an update trigger.
- Primary keys are `UUID` generated with `gen_random_uuid()`.
- Foreign keys always have explicit `ON DELETE` behavior defined.

---

## JSONB Column Conventions

JSONB schemas must be documented in `supabase/types/` as TypeScript-style type definitions (for cross-layer reference). Example:

```typescript
// supabase/types/schema.ts
type ColumnDefinition = {
  name: string;           // destination column name
  type: "string" | "integer" | "float" | "date" | "boolean";
  nullable: boolean;
  validation_rules?: ValidationRule[];
};

type SchemaRecord = {
  version: number;        // increment on breaking changes
  columns: ColumnDefinition[];
};
```

- Always include a `version` field in top-level JSONB objects to support migrations without DDL.
- Arrays inside JSONB are ordered. Order is semantically significant for transformation chains.
- Never query inside deeply nested JSONB in hot paths — denormalize frequently-queried fields into typed columns.

---

## Migration Rules

- All schema changes are managed via Supabase migration files (`supabase/migrations/`).
- Migration files are named `YYYYMMDDHHMMSS_descriptive_name.sql`.
- Migrations are **additive only** in production: add columns (nullable), add tables, add indexes. Never drop or rename in a single migration without a multi-phase deprecation plan.
- JSONB `version` bumps are handled in application code, not DDL.
- Run `supabase db diff` before writing a migration to verify the expected delta.

## Row-Level Security

- RLS is **enabled on all tables**. No table is left without a policy.
- Default posture: deny all. Policies explicitly grant access.
- Standard pattern:
  ```sql
  -- Users can only access their own rows
  CREATE POLICY "user_isolation" ON projects
    FOR ALL USING (auth.uid() = user_id);
  ```
- Service role operations (ETL callbacks, job runners) use the `service_role` key only in server-side contexts (FastAPI backend). Never expose the service role key to the frontend.
- RLS policies are tested via `supabase/tests/rls/` SQL test files.

## Storage

- **Bucket structure:**
  ```
  uploads/          # Raw user-uploaded files (private)
    {user_id}/
      {project_id}/
        {upload_id}.{ext}

  exports/          # Processed output files (private)
    {user_id}/
      {project_id}/
        {job_id}.{ext}
  ```
- All buckets are **private**. Frontend access is via short-lived presigned URLs generated by the FastAPI backend.
- Object paths are stored in the `uploads.storage_path` column — the ETL engine resolves them to presigned download URLs at job start.
- Maximum object size enforced at the bucket policy level (configurable; default 30MB per file).
- Completed export files are retained for 7 days, then eligible for lifecycle deletion.

---

## Indexes

- Index all foreign key columns.
- Index `jobs.status` and `job_files.status` — these are queried frequently in status polls.
- Index `uploads.project_id` and `schemas.project_id`.
- Use partial indexes for active records where appropriate:
  ```sql
  CREATE INDEX jobs_active_idx ON jobs (user_id, created_at DESC)
    WHERE status IN ('queued', 'running');
  ```

---

## Seed Data

- `supabase/seed.sql` contains only structural seed data (e.g., system configuration rows).
- No personal data or business-domain-specific data in seeds.
- Test fixtures live in `supabase/tests/fixtures/` and are scoped to the test runner.
